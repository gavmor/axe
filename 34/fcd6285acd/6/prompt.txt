9: ---
10: 
11: ## 1. Purpose
12: 
13: When an LLM refuses a request (e.g., "I cannot help with that"), axe currently treats it as a normal successful completion. The `content` field contains the refusal text, `stop_reason` is `"stop"` or `"end_turn"`, and exit code is 0. There is no way to programmatically distinguish a refusal from a real answer without inspecting the content.
14: 
15: Orchestration systems (like Agent Engine) need to detect refusals to mark runs appropriately, alert users, and potentially retry with different prompts. Centralizing this detection inside axe keeps the logic in one place rather than duplicated across every integration.
16: 
17: This milestone adds a `"refused"` boolean field to the `--json` output envelope. The field is always present when `--json` is used. It is `true` when the LLM response content matches common refusal patterns, `false` otherwise.
18: 
19: This milestone does **not** change exit codes, modify `stop_reason`, add a CLI flag to control refusal detection, or surface refusal status outside of `--json` output.
20: 
21: ---
22: 
23: ## 2. Design Decisions
24: 
25: The following decisions were made during planning and are binding for implementation:
26: 
27: 1. **Detection scope:** Refusal patterns are matched against the full response content, not just the beginning. Refusals sometimes come after a preamble (e.g., "I appreciate your question, however I cannot...").
28: 2. **Field presence:** The `"refused"` field is always present in `--json` output вЂ” both `true` and `false` values are emitted. It is never conditionally omitted. This provides a consistent schema for consumers.
29: 3. **Exit code unchanged:** A refusal is still a successful LLM call. The exit code remains 0. Orchestration systems use the `"refused"` field, not the exit code, to distinguish refusals.
30: 4. **`stop_reason` unchanged:** The original `stop_reason` from the LLM is preserved. No synthetic `"refusal"` stop reason is introduced.
31: 5. **No effect on non-JSON output:** When `--json` is not used, stdout contains only the raw LLM content. Refusal detection does not affect plain text output in any way. Axe output must remain safe to pipe.
32: 6. **Pattern-based, not exhaustive:** The detection uses a fixed set of common refusal patterns. False positives are less harmful than false negatives for this use case.
33: 7. **Case-insensitive matching:** All pattern matching is case-insensitive.
34: 8. **New package:** Refusal detection logic lives in a new `internal/refusal` package to keep it isolated and testable, separate from the CLI layer.
35: 9. **No new external dependencies.** Continue using stdlib only plus existing `spf13/cobra` and `BurntSushi/toml`.

39: ## 3. Requirements

41: ### 3.1 Refusal Detection Package (`internal/refusal/`)
42: 
43: **Requirement 1.1:** Create a new package `internal/refusal` with a single exported function:
44: 
45: ```go
46: func Detect(content string) bool
47: ```
48: 
49: - Returns `true` if the content matches any common LLM refusal pattern.
50: - Returns `false` if the content does not match any refusal pattern.
51: - Returns `false` for empty string input.
52: - Matching is case-insensitive.
53: - Matching is performed against the full content string (not just the beginning).
54: 
55: **Requirement 1.2:** The function must check for the following refusal patterns (case-insensitive substring matching):
56: 
57: | Pattern | Rationale |
58: |---------|-----------|
59: | `"i cannot"` | Direct refusal |
60: | `"i can't"` | Contraction form of "I cannot" |
61: | `"i'm unable to"` | Polite refusal |
62: | `"i am unable to"` | Expanded form |
63: | `"i'm not able to"` | Alternative phrasing |
64: | `"i am not able to"` | Expanded form |
65: | `"i must decline"` | Formal refusal |
66: | `"i don't have the ability"` | Capability-based refusal |
67: | `"i do not have the ability"` | Expanded form |
68: 
69: **Requirement 1.3:** In addition to the simple substring patterns above, the function must detect the compound pattern where the content contains `"as an ai"` AND also contains at least one of these refusal indicators (case-insensitive):
70: 
71: - `"i cannot"`
72: - `"i can't"`
73: - `"i'm unable"`
74: - `"i am unable"`
75: - `"i'm not able"`
76: - `"i am not able"`
77: - `"i must decline"`
78: 
79: The compound `"as an ai"` pattern only triggers a refusal when **both** the `"as an ai"` marker and a refusal indicator are present. `"as an ai"` alone is not a refusal.
80: 
81: **Requirement 1.4:** The detection function must use `strings.ToLower` for case normalization and `strings.Contains` for substring matching. No regular expressions.
82: 
83: **Requirement 1.5:** The detection function must not modify the input content. It is a pure read-only check.

167: | `"I'm not able to do that."` | `true` | Match: `"i'm not able to"`. |

171: | `"I do not have the ability to help."` | `true` | Match: `"i do not have the ability"`. |
172: | `"I CANNOT do that."` | `true` | Case-insensitive match. |
173: | `"i cannot do that."` | `true` | Case-insensitive match. |
174: | `"I'm sorry, but I cannot assist with that."` | `true` | Contains `"i cannot"` after preamble. |
175: | `"I appreciate your question. However, I'm unable to help."` | `true` | Contains `"i'm unable to"` after preamble. |
176: | `"As an AI, I cannot provide medical advice."` | `true` | Compound match: `"as an ai"` + `"i cannot"`. |
177: | `"As an AI, I'm happy to help with that!"` | `false` | Contains `"as an ai"` but no refusal indicator. |
178: | `"As an AI language model, I must decline."` | `true` | Compound match: `"as an ai"` + `"i must decline"`. |
179: | `"The user said 'I cannot do this' in their message."` | `true` | False positive вЂ” acceptable per design decision 6. The content contains `"i cannot"` as a substring. |
180: | `"Here's how you can't go wrong with this approach."` | `true` | False positive вЂ” acceptable per design decision 6. The content contains `"i can't"` as a substring. This is a known tradeoff: false positives are less harmful than false negatives. |
181: | `"I can notify you when it's ready."` | `false` | `"i can"` does not match `"i cannot"` or `"i can't"`. |
182: | `"Unable to connect to database."` | `false` | Does not contain `"i'm unable to"` or `"i am unable to"`. The `"i"` prefix is required. |
183: | `"The AI cannot process this."` | `false` | Contains `"cannot"` but not `"i cannot"`. Not a first-person refusal. |
184: | Content with only whitespace: `"   \n\t  "` | `false` | No pattern matches. |
185: | Very long content (10KB) with refusal phrase at the end | `true` | Full content scan detects it. |
186: | Very long content (10KB) with no refusal phrases | `false` | Full content scan finds nothing. |

23: ## 2. Design Decisions
24: 
25: The following decisions were made during planning and are binding for implementation:
26: 
27: 1. **Detection scope:** Refusal patterns are matched against the full response content, not just the beginning. Refusals sometimes come after a preamble (e.g., "I appreciate your question, however I cannot...").
28: 2. **Field presence:** The `"refused"` field is always present in `--json` output вЂ” both `true` and `false` values are emitted. It is never conditionally omitted. This provides a consistent schema for consumers.
29: 3. **Exit code unchanged:** A refusal is still a successful LLM call. The exit code remains 0. Orchestration systems use the `"refused"` field, not the exit code, to distinguish refusals.
30: 4. **`stop_reason` unchanged:** The original `stop_reason` from the LLM is preserved. No synthetic `"refusal"` stop reason is introduced.
31: 5. **No effect on non-JSON output:** When `--json` is not used, stdout contains only the raw LLM content. Refusal detection does not affect plain text output in any way. Axe output must remain safe to pipe.
32: 6. **Pattern-based, not exhaustive:** The detection uses a fixed set of common refusal patterns. False positives are less harmful than false negatives for this use case.
33: 7. **Case-insensitive matching:** All pattern matching is case-insensitive.
34: 8. **New package:** Refusal detection logic lives in a new `internal/refusal` package to keep it isolated and testable, separate from the CLI layer.
35: 9. **No new external dependencies.** Continue using stdlib only plus existing `spf13/cobra` and `BurntSushi/toml`.

39: ## 3. Requirements
40: 
41: ### 3.1 Refusal Detection Package (`internal/refusal/`)
42: 
43: **Requirement 1.1:** Create a new package `internal/refusal` with a single exported function:
44: 
45: ```go
46: func Detect(content string) bool
47: ```
48: 
49: - Returns `true` if the content matches any common LLM refusal pattern.
50: - Returns `false` if the content does not match any refusal pattern.
51: - Returns `false` for empty string input.
52: - Matching is case-insensitive.
53: - Matching is performed against the full content string (not just the beginning).
54: 
55: **Requirement 1.2:** The function must check for the following refusal patterns (case-insensitive substring matching):
56: 
57: | Pattern | Rationale |
58: |---------|-----------|
59: | `"i cannot"` | Direct refusal |
60: | `"i can't"` | Contraction form of "I cannot" |
61: | `"i'm unable to"` | Polite refusal |
62: | `"i am unable to"` | Expanded form |
63: | `"i'm not able to"` | Alternative phrasing |
64: | `"i am not able to"` | Expanded form |
65: | `"i must decline"` | Formal refusal |
66: | `"i don't have the ability"` | Capability-based refusal |
67: | `"i do not have the ability"` | Expanded form |
68: 
69: **Requirement 1.3:** In addition to the simple substring patterns above, the function must detect the compound pattern where the content contains `"as an ai"` AND also contains at least one of these refusal indicators (case-insensitive):
70: 
71: - `"i cannot"`
72: - `"i can't"`
73: - `"i'm unable"`
74: - `"i am unable"`
75: - `"i'm not able"`
76: - `"i am not able"`
77: - `"i must decline"`
78: 
79: The compound `"as an ai"` pattern only triggers a refusal when **both** the `"as an ai"` marker and a refusal indicator are present. `"as an ai"` alone is not a refusal.
80: 
81: **Requirement 1.4:** The detection function must use `strings.ToLower` for case normalization and `strings.Contains` for substring matching. No regular expressions.
82: 
83: **Requirement 1.5:** The detection function must not modify the input content. It is a pure read-only check.
84: 
85: ### 3.2 JSON Output Envelope (`cmd/run.go`)
86: 
87: **Requirement 2.1:** When the `--json` flag is set, the JSON output envelope must include a `"refused"` field of type boolean. The field must always be present (never omitted).
88: 
89: **Requirement 2.2:** The `"refused"` field value is the return value of `refusal.Detect(resp.Content)` where `resp.Content` is the final LLM response content.
90: 
91: **Requirement 2.3:** The field must appear in the envelope alongside existing fields. The complete `--json` envelope schema after this change:
92: 
93: ```json
94: {
95:   "model": "<string>",
96:   "content": "<string>",
97:   "input_tokens": "<int>",
98:   "output_tokens": "<int>",
99:   "stop_reason": "<string>",
100:   "duration_ms": "<int>",
101:   "tool_calls": "<int>",
102:   "tool_call_details": "<array>",
103:   "refused": "<bool>"
104: }
105: ```
106: 
107: **Requirement 2.4:** The `"refused"` field must work correctly for both single-shot responses (no tools) and conversation loop responses (with tools). In both cases, the detection runs against the final `resp.Content`.
108: 
109: ### 3.3 Golden File Updates
110: 
111: **Requirement 3.1:** All golden JSON files in `cmd/testdata/golden/json/` must be updated to include the `"refused": false` field. This ensures golden file tests pass after the schema change.
112: 
113: **Requirement 3.2:** The golden file masking function `maskJSONOutput` in `cmd/golden_test.go` does not require changes вЂ” the `"refused"` field is deterministic (always `false` for normal mock responses) and does not need masking.

155: ## 5. Edge Cases
156: 
157: ### 5.1 Content Patterns
158: 
159: | Scenario | `Detect()` returns | Rationale |
160: |----------|--------------------|-----------|
161: | `""` (empty string) | `false` | Nothing to match. |
162: | `"Hello, how can I help?"` | `false` | Normal response. |
163: | `"I cannot assist with that request."` | `true` | Direct refusal match: `"i cannot"`. |
164: | `"I can't help with that."` | `true` | Contraction match: `"i can't"`. |
165: | `"I'm unable to process this request."` | `true` | Match: `"i'm unable to"`. |
166: | `"I am unable to fulfill this."` | `true` | Match: `"i am unable to"`. |
167: | `"I'm not able to do that."` | `true` | Match: `"i'm not able to"`. |
168: | `"I am not able to do that."` | `true` | Match: `"i am not able to"`. |
169: | `"I must decline this request."` | `true` | Match: `"i must decline"`. |
170: | `"I don't have the ability to do that."` | `true` | Match: `"i don't have the ability"`. |
171: | `"I do not have the ability to help."` | `true` | Match: `"i do not have the ability"`. |
172: | `"I CANNOT do that."` | `true` | Case-insensitive match. |
173: | `"i cannot do that."` | `true` | Case-insensitive match. |
174: | `"I'm sorry, but I cannot assist with that."` | `true` | Contains `"i cannot"` after preamble. |
175: | `"I appreciate your question. However, I'm unable to help."` | `true` | Contains `"i'm unable to"` after preamble. |
176: | `"As an AI, I cannot provide medical advice."` | `true` | Compound match: `"as an ai"` + `"i cannot"`. |
177: | `"As an AI, I'm happy to help with that!"` | `false` | Contains `"as an ai"` but no refusal indicator. |
178: | `"As an AI language model, I must decline."` | `true` | Compound match: `"as an ai"` + `"i must decline"`. |
179: | `"The user said 'I cannot do this' in their message."` | `true` | False positive вЂ” acceptable per design decision 6. The content contains `"i cannot"` as a substring. |
180: | `"Here's how you can't go wrong with this approach."` | `true` | False positive вЂ” acceptable per design decision 6. The content contains `"i can't"` as a substring. This is a known tradeoff: false positives are less harmful than false negatives. |
181: | `"I can notify you when it's ready."` | `false` | `"i can"` does not match `"i cannot"` or `"i can't"`. |
182: | `"Unable to connect to database."` | `false` | Does not contain `"i'm unable to"` or `"i am unable to"`. The `"i"` prefix is required. |
183: | `"The AI cannot process this."` | `false` | Contains `"cannot"` but not `"i cannot"`. Not a first-person refusal. |
184: | Content with only whitespace: `"   \n\t  "` | `false` | No pattern matches. |
185: | Very long content (10KB) with refusal phrase at the end | `true` | Full content scan detects it. |
186: | Very long content (10KB) with no refusal phrases | `false` | Full content scan finds nothing. |

189: 
190: | Scenario | `"refused"` value | Notes |
191: |----------|-------------------|-------|
192: | Normal LLM response | `false` | Standard case. |
193: | LLM returns refusal text | `true` | Detection triggered. |
194: | LLM response with tools, final text is refusal | `true` | Detection runs on the final `resp.Content`. |
195: | LLM response with tools, final text is normal | `false` | Detection runs on the final `resp.Content`. |

197: | Single-shot (no tools), refusal response | `true` | Same codepath. |

213: ## 6. Constraints
214: 
215: **Constraint 1:** No new external dependencies. `go.mod` must still contain only `spf13/cobra` and `BurntSushi/toml` as direct dependencies.
216: 
217: **Constraint 2:** No regular expressions in the detection function. Use `strings.ToLower` + `strings.Contains` only.
218: 
219: **Constraint 3:** Detection is read-only. The `Detect` function must not modify the input string or have any side effects.
220: 
221: **Constraint 4:** The `"refused"` field is only present in `--json` output. It must not appear in plain text stdout, stderr, verbose output, or dry-run output.
222: 
223: **Constraint 5:** The `"refused"` field must always be present in `--json` output (never conditionally omitted). The value is always a JSON boolean (`true` or `false`).
224: 
225: **Constraint 6:** Refusal detection does not affect exit codes. A detected refusal still exits 0.
226: 
227: **Constraint 7:** Refusal detection does not affect memory. If memory is enabled, the refusal response is still appended as a normal entry.
228: 
229: **Constraint 8:** The refusal pattern list is hard-coded. It is not user-configurable, not loaded from TOML, and not loaded from a skill file.
230: 
231: **Constraint 9:** Cross-platform compatibility: must build and run on Linux, macOS, and Windows. Only stdlib string operations are used.
