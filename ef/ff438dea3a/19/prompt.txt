11: ## 1. Purpose
12: 
13: Implement the `url_fetch` tool — a built-in tool registered in the `Registry` that performs an HTTP GET request and returns the response body as text. This tool is needed for feature parity when axe replaces the custom container worker in Agent Engine.
14: 
15: This tool follows the same pattern as the M3-M7 tools (`list_directory`, `read_file`, `write_file`, `edit_file`, `run_command`):
16: 
17: - **`RegisterAll`** — extended with one additional `r.Register(...)` call
18: - **`toolname.URLFetch`** — new constant declared in `internal/toolname/toolname.go`
19: - **`ExecContext`** — reused for `Stderr` and `Verbose` (verbose logging). `Workdir` is not used by this tool.
20: 
21: This tool does NOT use `validatePath` or `isWithinDir`. It operates on URLs, not file paths. There is no filesystem interaction.

24: 
25: ## 2. Design Decisions
26: 
27: The following decisions were made during planning and are binding for implementation:
28: 
29: 1. **HTTP GET only.** The tool performs a single HTTP GET request. No POST, PUT, DELETE, or other methods. No request body. No custom headers.
30: 
31: 2. **Scheme validation.** Only `http://` and `https://` schemes are accepted. Any other scheme (e.g., `file://`, `ftp://`, `data:`) is rejected before any network request is made. This prevents the LLM from reading local files or using unexpected protocols.
32: 
33: 3. **URL parsing via `net/url.Parse`.** The `url` argument is parsed using Go's `net/url.Parse()`. If parsing fails, the tool returns an error. The parsed URL's `Scheme` field is checked against `"http"` and `"https"`.
34: 
35: 4. **Context-based timeout.** The tool creates an `http.Request` using `http.NewRequestWithContext(ctx, "GET", url, nil)`. The `ctx` parameter from the executor signature is passed directly. This means the tool inherits whatever timeout the caller provides (from the agent's `--timeout` flag). The tool does not impose its own timeout.
36: 
37: 5. **`http.DefaultClient` for execution.** The tool uses `http.DefaultClient.Do(req)` to execute the request. No custom transport, no TLS configuration, no cookie jar. `http.DefaultClient` follows redirects (up to 10 by default per Go stdlib).
38: 
39: 6. **Response body read limit via `io.LimitReader`.** The response body is read using `io.LimitReader(resp.Body, maxReadBytes+1)` where `maxReadBytes` is 10000. Reading one byte beyond the limit allows detecting whether truncation occurred. If exactly `maxReadBytes+1` bytes are read, the response is truncated to `maxReadBytes` bytes and a truncation notice is appended.
40: 
41: 7. **Truncation notice.** When the response exceeds 10000 bytes, the content is truncated to 10000 bytes and the string `"\n... [response truncated, exceeded 10000 characters]"` is appended. Truncation is byte-level, not rune-aware.
42: 
43: 8. **Non-2xx status codes return an error.** If the HTTP response status code is outside the 200-299 range, the tool returns `ToolResult{IsError: true}` with content in the format `"HTTP %d: %s"` where the first value is the status code and the second value is the response body (truncated if needed). The response body is still read and included because it often contains useful error information.
44: 
45: 9. **Success result.** For 2xx responses, the tool returns `ToolResult{CallID: call.ID, Content: string(body), IsError: false}` where `body` is the response body after truncation if applicable.
46: 
47: 10. **`resp.Body.Close()` via `defer`.** The response body is closed via `defer resp.Body.Close()` immediately after a successful `http.DefaultClient.Do()` call.
48: 
49: 11. **Network and DNS errors.** If `http.DefaultClient.Do()` returns an error (DNS failure, connection refused, TLS error, context cancellation), the tool returns `ToolResult{CallID: call.ID, Content: err.Error(), IsError: true}`.
50: 
51: 12. **Single parameter: `url`.** The tool has one required parameter named `url` of type `string`. There are no optional parameters. An empty or missing `url` is rejected before any parsing or network request.
52: 
53: 13. **`call_agent` remains outside the registry.** Same as all prior tool milestones.
54: 
55: 14. **No new external dependencies.** Uses only Go stdlib (`net/http`, `net/url`, `io`, `fmt`, `context`).

170: ## 5. Edge Cases

172: ### 5.1 URL Argument
173: 
174: | Scenario | Input `url` | Behavior |
175: |----------|------------|----------|
176: | Empty URL | `""` | Error: `"url is required"` |
177: | Missing `url` key in Arguments map | Key absent | `call.Arguments["url"]` returns `""` -> Error: `"url is required"` |
178: | Valid HTTPS URL | `"https://example.com"` | Normal HTTP GET. Returns response body. |
179: | Valid HTTP URL | `"http://example.com"` | Normal HTTP GET. Returns response body. |
180: | URL without scheme | `"example.com"` | `net/url.Parse` succeeds but `Scheme` is empty. Error: `unsupported scheme "": only http and https are allowed`. |
181: | File scheme | `"file:///etc/passwd"` | Error: `unsupported scheme "file": only http and https are allowed`. |
182: | FTP scheme | `"ftp://files.example.com/data"` | Error: `unsupported scheme "ftp": only http and https are allowed`. |
183: | Data URI | `"data:text/plain;base64,SGVsbG8="` | Error: `unsupported scheme "data": only http and https are allowed`. |
184: | Malformed URL | `"://bad"` | `net/url.Parse` returns an error. Error includes parse error message. |
185: | URL with port | `"http://localhost:8080/path"` | Normal HTTP GET. Port is part of the URL. |
186: | URL with query string | `"https://api.example.com/data?key=val"` | Normal HTTP GET. Query string is preserved. |
187: | URL with fragment | `"https://example.com/page#section"` | Normal HTTP GET. Fragment is not sent to server (per HTTP spec). |
188: | URL with auth | `"https://user:pass@example.com"` | Normal HTTP GET. Go's http client includes auth in the request. |
189: | Very long URL | 10,000+ character URL | Passed to http client. Server may reject with 414. Tool reports the server's response. |

209: ### 5.3 Response Body Truncation
210: 
211: | Scenario | Response size | Behavior |
212: |----------|--------------|----------|
213: | Response < 10000 bytes | < 10000 bytes | No truncation. Content is full body. |
214: | Response = 10000 bytes exactly | 10000 bytes | No truncation. The threshold is "exceeds", not "equals". (`io.LimitReader` reads 10001 max; if read returns exactly 10000 bytes, no truncation.) |
215: | Response > 10000 bytes | > 10000 bytes | Truncated to 10000 bytes. `"\n... [response truncated, exceeded 10000 characters]"` appended. |
216: | Large response on success (2xx) | > 10000 bytes, status 200 | Content: first 10000 bytes + truncation notice. `IsError: false`. |
217: | Large response on error (non-2xx) | > 10000 bytes, status 500 | Content: `"HTTP 500: "` + first 10000 bytes + truncation notice. `IsError: true`. |
218: | Truncation at multi-byte boundary | Truncated mid-UTF-8 sequence | Truncation occurs at the byte level. The resulting string may contain an incomplete UTF-8 sequence. No rune-aware truncation is performed. |
219: | Empty response body | 0 bytes | Content is empty string (or `"HTTP <code>: "` for non-2xx). |
220: | Binary response | Binary content | Content contains raw bytes as string. No encoding detection or conversion. |

435: ## 11. Notes
436: 
437: - **No path validation needed.** Unlike file tools, `url_fetch` operates on URLs, not file paths. The scheme validation (`http`/`https` only) is the security boundary, preventing `file://` access to the local filesystem.
438: - **`io.LimitReader` for memory safety.** Without a read limit, the LLM could instruct the tool to fetch a multi-gigabyte resource, causing OOM. `io.LimitReader` caps memory usage at `maxReadBytes+1` bytes regardless of response size. The extra byte is used to detect truncation without requiring a second read or `Content-Length` header inspection.
439: - **`http.DefaultClient` follows redirects.** Go's default client follows up to 10 redirects. This is desirable behavior — the LLM doesn't need to handle redirects manually. If a redirect loop occurs, `Do()` returns an error which the tool reports.
440: - **Byte-level truncation, not rune-aware.** Consistent with `run_command` (100KB byte truncation). For web content (predominantly ASCII/UTF-8), byte-level truncation rarely breaks in practice. The truncation notice clearly indicates truncation occurred.
441: - **Non-2xx body is included in error.** Many APIs return useful error information in the response body (JSON error objects, HTML error pages). Including the body (truncated) in the error result gives the LLM actionable information.
442: - **No `Workdir` usage.** Unlike file tools, `url_fetch` does not use `ExecContext.Workdir`. The tool operates on network URLs, not filesystem paths. `Workdir` is irrelevant.
443: - **The `RegisterAll` pattern continues.** This adds the sixth tool: `list_directory`, `read_file`, `write_file`, `edit_file`, `run_command`, `url_fetch`.
444: - **Security model.** The `url_fetch` tool allows the LLM to make arbitrary HTTP GET requests to any `http://` or `https://` URL. This is by design — if an agent has `tools = ["url_fetch"]` in its TOML config, the user has explicitly opted in. Scheme validation prevents `file://` abuse. Network-level restrictions (firewalls, egress policies) are outside axe's scope.

24: 
25: ## 2. Design Decisions
26: 
27: The following decisions were made during planning and are binding for implementation:
28: 
29: 1. **HTTP GET only.** The tool performs a single HTTP GET request. No POST, PUT, DELETE, or other methods. No request body. No custom headers.
30: 
31: 2. **Scheme validation.** Only `http://` and `https://` schemes are accepted. Any other scheme (e.g., `file://`, `ftp://`, `data:`) is rejected before any network request is made. This prevents the LLM from reading local files or using unexpected protocols.
32: 
33: 3. **URL parsing via `net/url.Parse`.** The `url` argument is parsed using Go's `net/url.Parse()`. If parsing fails, the tool returns an error. The parsed URL's `Scheme` field is checked against `"http"` and `"https"`.
34: 
35: 4. **Context-based timeout.** The tool creates an `http.Request` using `http.NewRequestWithContext(ctx, "GET", url, nil)`. The `ctx` parameter from the executor signature is passed directly. This means the tool inherits whatever timeout the caller provides (from the agent's `--timeout` flag). The tool does not impose its own timeout.
36: 
37: 5. **`http.DefaultClient` for execution.** The tool uses `http.DefaultClient.Do(req)` to execute the request. No custom transport, no TLS configuration, no cookie jar. `http.DefaultClient` follows redirects (up to 10 by default per Go stdlib).
38: 
39: 6. **Response body read limit via `io.LimitReader`.** The response body is read using `io.LimitReader(resp.Body, maxReadBytes+1)` where `maxReadBytes` is 10000. Reading one byte beyond the limit allows detecting whether truncation occurred. If exactly `maxReadBytes+1` bytes are read, the response is truncated to `maxReadBytes` bytes and a truncation notice is appended.
40: 
41: 7. **Truncation notice.** When the response exceeds 10000 bytes, the content is truncated to 10000 bytes and the string `"\n... [response truncated, exceeded 10000 characters]"` is appended. Truncation is byte-level, not rune-aware.
42: 
43: 8. **Non-2xx status codes return an error.** If the HTTP response status code is outside the 200-299 range, the tool returns `ToolResult{IsError: true}` with content in the format `"HTTP %d: %s"` where the first value is the status code and the second value is the response body (truncated if needed). The response body is still read and included because it often contains useful error information.
44: 
45: 9. **Success result.** For 2xx responses, the tool returns `ToolResult{CallID: call.ID, Content: string(body), IsError: false}` where `body` is the response body after truncation if applicable.
46: 
47: 10. **`resp.Body.Close()` via `defer`.** The response body is closed via `defer resp.Body.Close()` immediately after a successful `http.DefaultClient.Do()` call.
48: 
49: 11. **Network and DNS errors.** If `http.DefaultClient.Do()` returns an error (DNS failure, connection refused, TLS error, context cancellation), the tool returns `ToolResult{CallID: call.ID, Content: err.Error(), IsError: true}`.
50: 
51: 12. **Single parameter: `url`.** The tool has one required parameter named `url` of type `string`. There are no optional parameters. An empty or missing `url` is rejected before any parsing or network request.
52: 
53: 13. **`call_agent` remains outside the registry.** Same as all prior tool milestones.
54: 
55: 14. **No new external dependencies.** Uses only Go stdlib (`net/http`, `net/url`, `io`, `fmt`, `context`).

213: | Response < 10000 bytes | < 10000 bytes | No truncation. Content is full body. |
214: | Response = 10000 bytes exactly | 10000 bytes | No truncation. The threshold is "exceeds", not "equals". (`io.LimitReader` reads 10001 max; if read returns exactly 10000 bytes, no truncation.) |
215: | Response > 10000 bytes | > 10000 bytes | Truncated to 10000 bytes. `"\n... [response truncated, exceeded 10000 characters]"` appended. |
216: | Large response on success (2xx) | > 10000 bytes, status 200 | Content: first 10000 bytes + truncation notice. `IsError: false`. |
217: | Large response on error (non-2xx) | > 10000 bytes, status 500 | Content: `"HTTP 500: "` + first 10000 bytes + truncation notice. `IsError: true`. |
218: | Truncation at multi-byte boundary | Truncated mid-UTF-8 sequence | Truncation occurs at the byte level. The resulting string may contain an incomplete UTF-8 sequence. No rune-aware truncation is performed. |
219: | Empty response body | 0 bytes | Content is empty string (or `"HTTP <code>: "` for non-2xx). |
220: | Binary response | Binary content | Content contains raw bytes as string. No encoding detection or conversion. |

224: | Scenario | Behavior |
225: |----------|----------|
226: | DNS resolution failure | `Do()` returns an error. `IsError: true`. Content: error message (e.g., `"dial tcp: lookup nonexistent.invalid: no such host"`). |
227: | Connection refused | `Do()` returns an error. `IsError: true`. Content: error message. |
228: | Connection timeout | `Do()` returns an error (via context deadline). `IsError: true`. Content: error message. |
229: | TLS certificate error | `Do()` returns an error. `IsError: true`. Content: error message (e.g., `"tls: certificate is not trusted"`). |
230: | Context cancelled before request | `Do()` returns context error immediately. `IsError: true`. Content: `"context canceled"` or similar. |
231: | Context cancelled during response read | `io.ReadAll` returns a context error. `IsError: true`. Content: error message. |
232: | Server closes connection mid-response | `io.ReadAll` returns an error (e.g., `"unexpected EOF"`). `IsError: true`. Content: error message. |
233: | Redirect loop (> 10 hops) | `Do()` returns an error. `IsError: true`. Content: error message about redirect loop. |

435: ## 11. Notes
436: 
437: - **No path validation needed.** Unlike file tools, `url_fetch` operates on URLs, not file paths. The scheme validation (`http`/`https` only) is the security boundary, preventing `file://` access to the local filesystem.
438: - **`io.LimitReader` for memory safety.** Without a read limit, the LLM could instruct the tool to fetch a multi-gigabyte resource, causing OOM. `io.LimitReader` caps memory usage at `maxReadBytes+1` bytes regardless of response size. The extra byte is used to detect truncation without requiring a second read or `Content-Length` header inspection.
439: - **`http.DefaultClient` follows redirects.** Go's default client follows up to 10 redirects. This is desirable behavior — the LLM doesn't need to handle redirects manually. If a redirect loop occurs, `Do()` returns an error which the tool reports.
440: - **Byte-level truncation, not rune-aware.** Consistent with `run_command` (100KB byte truncation). For web content (predominantly ASCII/UTF-8), byte-level truncation rarely breaks in practice. The truncation notice clearly indicates truncation occurred.
441: - **Non-2xx body is included in error.** Many APIs return useful error information in the response body (JSON error objects, HTML error pages). Including the body (truncated) in the error result gives the LLM actionable information.
442: - **No `Workdir` usage.** Unlike file tools, `url_fetch` does not use `ExecContext.Workdir`. The tool operates on network URLs, not filesystem paths. `Workdir` is irrelevant.
443: - **The `RegisterAll` pattern continues.** This adds the sixth tool: `list_directory`, `read_file`, `write_file`, `edit_file`, `run_command`, `url_fetch`.
444: - **Security model.** The `url_fetch` tool allows the LLM to make arbitrary HTTP GET requests to any `http://` or `https://` URL. This is by design — if an agent has `tools = ["url_fetch"]` in its TOML config, the user has explicitly opted in. Scheme validation prevents `file://` abuse. Network-level restrictions (firewalls, egress policies) are outside axe's scope.
