<content>1: # Specification: MCP (Model Context Protocol) Tool Support
2: 
3: **Status:** Draft
4: **Version:** 1.0
5: **Created:** 2026-03-05
6: **GitHub Issue:** https://github.com/jrswab/axe/issues/9
7: **Scope:** MCP client support — connect to external MCP servers, discover tools, and route LLM tool calls to them
8: 
9: ---
10: 
11: ## 1. Purpose
12: 
13: Add support for connecting to MCP servers as a tool source, allowing agents to use tools
14: exposed by external MCP servers alongside axe's built-in tools.
15: 
16: Axe acts as an **MCP client only** (tool consumer). Exposing axe's own tools as an MCP
17: server is out of scope.
18: 
19: An agent declares MCP servers in its TOML config. At startup, axe connects to each
20: declared server, discovers available tools via the MCP `tools/list` method, converts them
21: to `provider.Tool` definitions, and appends them to the LLM request. During the
22: conversation loop, tool calls targeting MCP tools are routed to the appropriate MCP
23: server session via the MCP `tools/call` method.
24: 
25: This feature introduces one new external dependency:
26: `github.com/modelcontextprotocol/go-sdk` (the official MCP Go SDK).
27: 
28: ---
29: 
30: ## 2. Design Decisions
31: 
32: The following decisions were made during planning and are binding for implementation:
33: 
34: 1. **Official Go SDK.** Use `github.com/modelcontextprotocol/go-sdk/mcp` for MCP client
35:    functionality. This provides spec-compliant transport implementations
36:    (`StreamableClientTransport`, `SSEClientTransport`), session management, and
37:    JSON-RPC handling. Pin to a specific release version in `go.mod`.
38: 
39: 2. **Two transport types.** Support `"sse"` and `"streamable-http"` as transport values.
40:    `"sse"` uses the SDK's `SSEClientTransport`. `"streamable-http"` uses the SDK's
41:    `StreamableClientTransport`. No other transport types are supported.
42: 
43: 3. **Eager connection at startup.** All declared MCP servers are connected before the
44:    first LLM call. Tool discovery (`tools/list`) happens immediately after connection.
45:    Sessions remain open for the duration of the conversation loop and are closed when
46:    the agent run completes.
47: 
48: 4. **Fail fast on connection failure.** If any declared MCP server cannot be reached
49:    (connection refused, DNS failure, timeout, TLS error, initialization handshake
50:    failure), the agent fails immediately with exit code 3 (API error).
51: 
52: 5. **Runtime-only tool validation.** MCP tool names are discovered dynamically via
53:    `tools/list`. They are not added to `toolname.ValidNames()`. The `tools` field in
54:    agent TOML controls only built-in tools. MCP tools are controlled entirely by the
55:    `[[mcp_servers]]` configuration.
56: 
57: 6. **Built-in tool wins on name collision.** If an MCP server exposes a tool whose name
58:    matches a built-in tool that is enabled for this agent (present in `cfg.Tools`), the
59:    MCP tool is silently skipped. A warning is logged to stderr in verbose mode. The
60:    built-in tool takes precedence because it has known, tested behavior.
61: 
62: 7. **MCP-to-MCP collision is an error.** If two different MCP servers expose tools with
63:    the same name, the agent fails at startup with exit code 2 (config error). The error
64:    message names both MCP server names and the colliding tool name.
65: 
66: 8. **Flatten complex JSON Schema types.** MCP tools use full JSON Schema for
67:    `InputSchema`. Axe's `provider.ToolParameter` has `Type`, `Description`, and
68:    `Required` fields. Simple JSON Schema types (`string`, `integer`, `number`,
69:    `boolean`) map directly to the `Type` field. Complex types (`object`, `array`,
70:    `oneOf`, `anyOf`, `allOf`, or any type not in the simple set) are mapped to
71:    `Type: "string"` with the full schema description preserved in the `Description`
72:    field. The LLM reads the description and formats accordingly.
73: 
74: 9. **Concatenate text content from results.** MCP `CallToolResult` can contain multiple
75:    content items (`TextContent`, `ImageContent`, `EmbeddedResource`, etc.). All
76:    `TextContent` items are extracted and concatenated with newline (`"\n"`) separators.
77:    Non-text content items are ignored (axe is a text-only CLI tool). If no `TextContent`
78:    items exist in the result, the tool returns `ToolResult{IsError: true}` with content
79:    `"MCP tool returned no text content"`.
80: 
81: 10. **Pass arguments as strings.** Axe's `ToolCall.Arguments` is `map[string]string`.
82:     When dispatching to an MCP server, each string value is placed into a
83:     `map[string]any` as-is (the `any` value is the original `string`). The MCP server is
84:     responsible for any type coercion. This is consistent with how axe handles built-in
85:     tool arguments.
86: 
87: 11. **Environment variable interpolation for headers.** Header values support `${VAR}`
88:     syntax. The pattern `${VAR_NAME}` is replaced with `os.Getenv("VAR_NAME")`. If the
89:     referenced environment variable is not set or is empty, the agent fails at config
90:     resolution time with exit code 2 (config error). The error message names the missing
91:     variable and the MCP server name. Static header values (no `${}` pattern) pass
92:     through unchanged.
93: 
94: 12. **Dry-run shows config only, no connection.** In `--dry-run` mode, MCP server
95:     configuration (name, URL, transport) is displayed in a `--- MCP Servers ---` section.
96:     No connections are established. Tool names are not shown because they require a live
97:     connection.
98: 
99: 13. **Sub-agents get their own MCP connections.** When a sub-agent has `[[mcp_servers]]`
100:     in its own config, it establishes its own MCP connections independently. The parent
101:     agent's MCP connections are never shared with sub-agents. This preserves the
102:     sub-agent opacity principle.
103: 
104: 14. **MCP tool calls appear in `--json` output.** MCP tool calls are included in the
105:     `tool_call_details` array with the same structure as built-in tool calls: `name`,
106:     `input`, `output`, `is_error`. There is no distinction between MCP and built-in tool
107:     calls in the JSON envelope.
108: 
109: 15. **Connection timeout inherits `--timeout`.** MCP server connections use the same
110:     timeout context as the agent run. If the `--timeout` flag is 120 seconds, MCP
111:     connections and tool calls share that deadline.

117: ### 3.1 Agent Config: `MCPServerConfig` Struct
118: 
119: **Requirement 1.1:** Add a new exported struct `MCPServerConfig` to
120: `internal/agent/agent.go` with the following fields:
121: 
122: | Field | Go Type | TOML Tag | Description |
123: |-------|---------|----------|-------------|
124: | `Name` | `string` | `toml:"name"` | Human-readable identifier for this MCP server |
125: | `URL` | `string` | `toml:"url"` | The MCP server endpoint URL |
126: | `Transport` | `string` | `toml:"transport"` | Transport type: `"sse"` or `"streamable-http"` |
127: | `Headers` | `map[string]string` | `toml:"headers"` | Optional HTTP headers (supports `${VAR}` interpolation) |

129: **Requirement 1.2:** Add a field `MCPServers []MCPServerConfig` with tag
130: `toml:"mcp_servers"` to the `AgentConfig` struct.
131: 
132: **Requirement 1.3:** The TOML syntax for declaring MCP servers is `[[mcp_servers]]`
133: (array of tables). Example:
134: 
135: ```toml
136: [[mcp_servers]]
137: name = "my-tools"
138: url = "https://my-mcp-server.example.com/sse"
139: transport = "sse"
140: 
141: [[mcp_servers]]
142: name = "pipedream"
143: url = "https://remote.mcp.pipedream.net"
144: transport = "streamable-http"
145: headers = { Authorization = "Bearer ${PIPEDREAM_TOKEN}" }
146: ```

149:

163: have the same `Name` (case-sensitive, after trimming whitespace), return error:
164: `"mcp_servers: duplicate server name %q"`.
165: 
166: **Requirement 2.4:** Validation does NOT check URL format, header values, or environment
167: variable availability. Those are runtime concerns resolved at connection time.

169: ### 3.3 Environment Variable Interpolation
170: 
171: **Requirement 3.1:** Create a new package `internal/envinterp/` with a single file
172: `envinterp.go`.
173: 
174: **Requirement 3.2:** Export a function
175: `ExpandHeaders(headers map[string]string) (map[string]string, error)`.
176: 
177: **Requirement 3.3:** The function iterates over each key-value pair in the input map. For
178: each value, it scans for patterns matching the regex `\$\{([^}]+)\}`. For each match:
179: - Extract the variable name (the content between `${` and `}`).
180: - Look up the value via `os.Getenv(varName)`.
181: - If the environment variable is not set or is empty, return an error:
182:   `"environment variable %q is not set"`.
183: - Replace the `${VAR_NAME}` pattern with the environment variable value.
184: 
185: **Requirement 3.4:** Multiple `${VAR}` patterns in a single header value are all
186: expanded. Example: `"${SCHEME}://${HOST}"` with `SCHEME=https` and `HOST=example.com`
187: produces `"https://example.com"`.
188: 
189: **Requirement 3.5:** Values with no `${...}` patterns are returned unchanged (static
190: values).
191: 
192: **Requirement 3.6:** If the input map is nil, return nil and no error.
193: 
194: **Requirement 3.7:** The returned map is a new map instance. The input map is not
195: modified.

197: ### 3.4 MCP Client Package
198: 
199: **Requirement 4.1:** Create a new package `internal/mcpclient/` with a file
200: `mcpclient.go`.
201: 
202: **Requirement 4.2:** Export a struct `Client` that wraps an MCP client session. The
203: struct holds:
204: - The server name (from `MCPServerConfig.Name`)
205: - The SDK `*mcp.ClientSession`
206: 
207: **Requirement 4.3:** Export a function
208: `Connect(ctx context.Context, cfg agent.MCPServerConfig) (*Client, error)`.
209: 
210: This function:
211: 1. Expands environment variables in `cfg.Headers` via `envinterp.ExpandHeaders()`.
212:    If expansion fails, returns the error (caller maps to exit code 2).
213: 2. Creates an `*http.Client` with a custom `http.RoundTripper` that injects the
214:    expanded headers into every outgoing HTTP request.
215: 3. Based on `cfg.Transport`:
216:    - `"sse"`: Creates an `mcp.SSEClientTransport` with the custom HTTP client and
217:      `cfg.URL` as the endpoint.
218:    - `"streamable-http"`: Creates an `mcp.StreamableClientTransport` with the custom
219:      HTTP client and `cfg.URL` as the endpoint.
220: 4. Creates an `mcp.Client` with implementation info `{Name: "axe", Version: <axe version>}`.
221: 5. Calls `client.Connect(ctx, transport)` to establish the session (performs the MCP
222:    initialize/initialized handshake).
223: 6. Returns the wrapped `*Client` or an error if connection fails.
224: 
225: **Requirement 4.4:** Export a method
226: `(c *Client) ListTools(ctx context.Context) ([]provider.Tool, error)`.
227: 
228: This method:
229: 1. Calls `session.ListTools(ctx, nil)` to get the MCP `ListToolsResult`.
230: 2. Converts each `mcp.Tool` to a `provider.Tool`:
231:    - `Name`: from `mcp.Tool.Name`.
232:    - `Description`: from `mcp.Tool.Description`.
233:    - `Parameters`: Extracted from `mcp.Tool.InputSchema`. The `InputSchema` from the
234:      client side is a `map[string]any` representing a JSON Schema object. Extract the
235:      `"properties"` key (expected to be `map[string]any`). For each property, extract
236:      `"type"` and `"description"`. The `"required"` array from the top-level schema
237:      determines which parameters are required.
238:    - **Type mapping for parameters:**
239:      - `"string"`, `"integer"`, `"number"`, `"boolean"` map directly to the
240:        `ToolParameter.Type` field.
241:      - Any other type value (including `"object"`, `"array"`, empty, missing, or
242:        composite schemas like `oneOf`) maps to `Type: "string"`. The original type
243:        information is prepended to the description:
244:        `"(JSON Schema type: <original_type>) <description>"`.
245:      - If the `"type"` key is missing from a property, use `Type: "string"`.
246: 3. Returns the converted slice or an error.
247: 
248: **Requirement 4.5:** Export a method
249: `(c *Client) CallTool(ctx context.Context, call provider.ToolCall) (provider.ToolResult, error)`.
250: 
251: This method:
252: 1. Converts `call.Arguments` (`map[string]string`) to `map[string]any` by assigning
253:    each string value as-is.
254: 2. Creates `mcp.CallToolParams{Name: call.Name, Arguments: args}`.
255: 3. Calls `session.CallTool(ctx, params)`.
256: 4. Converts the result:
257:    - Iterates over `result.Content`. For each item, checks if it is `*mcp.TextContent`.
258:      If so, appends the `Text` field to a slice.
259:    - Joins all collected text items with `"\n"`.
260:    - If no text items are found, returns
261:      `ToolResult{CallID: call.ID, Content: "MCP tool returned no text content", IsError: true}`.
262:    - If `result.IsError` is true, returns
263:      `ToolResult{CallID: call.ID, Content: <joined text>, IsError: true}`.
264:    - Otherwise returns
265:      `ToolResult{CallID: call.ID, Content: <joined text>, IsError: false}`.
266: 
267: **Requirement 4.6:** Export a method `(c *Client) Name() string` that returns the server
268: name.
269: 
270: **Requirement 4.7:** Export a method `(c *Client) Close() error` that closes the MCP
271: session.

273: ### 3.5 MCP Tool Router
274: 
275: **Requirement 5.1:** Create a file `internal/mcpclient/router.go`.
276: 
277: **Requirement 5.2:** Export a struct `Router` that maps tool names to their owning
278: `*Client`.
279: 
280: **Requirement 5.3:** Export a function `NewRouter() *Router` that returns an initialized
281: router with an empty map.
282: 
283: **Requirement 5.4:** Export a method
284: `(r *Router) Register(client *Client, tools []provider.Tool, builtinNames map[string]bool) ([]provider.Tool, error)`.
285: 
286: This method:
287: 1. For each tool in `tools`:
288:    - If the tool name exists in `builtinNames`, skip it (built-in wins). If verbose
289:      logging is needed, the caller handles that — this method is silent on skips but
290:      returns only the non-skipped tools.
291:    - If the tool name is already registered in the router from a different client,
292:      return an error:
293:      `"MCP tool name collision: tool %q is exposed by both %q and %q"` where the two
294:      values are the server names.
295:    - Otherwise, add the mapping from tool name to client.
296: 2. Returns the filtered slice of `provider.Tool` definitions (excluding skipped tools)
297:    and nil error on success.
298: 
299: **Requirement 5.5:** Export a method
300: `(r *Router) Dispatch(ctx context.Context, call provider.ToolCall) (provider.ToolResult, error)`.
301: 
302: This method:
303: 1. Looks up the client by `call.Name`.
304: 2. If not found, returns `(zero ToolResult, error)` with
305:    `"unknown MCP tool %q"`.
306: 3. Delegates to `client.CallTool(ctx, call)`.
307: 
308: **Requirement 5.6:** Export a method `(r *Router) Has(name string) bool` that returns
309: true if the tool name is registered in the router.
310: 
311: **Requirement 5.7:** Export a method `(r *Router) Close() error` that calls `Close()` on
312: every unique client. Each client is closed exactly once even if it registered multiple
313: tools. Returns the first error encountered (or nil).

439: ## 5. Edge Cases
440: 
441: ### 5.1 Config Parsing
442: 
443: | Scenario | Input | Behavior |
444: |----------|-------|----------|
445: | No MCP servers | `mcp_servers` key absent | `cfg.MCPServers` is nil. No MCP logic executes. Agent runs normally. |
446: | Empty MCP servers array | `mcp_servers = []` | Same as absent. Zero-length slice. No MCP logic executes. |
447: | Missing name | `[[mcp_servers]]` with no `name` key | Validation error: `"mcp_servers[0]: name is required"`. Exit 2. |
448: | Missing URL | `name = "x"` but no `url` | Validation error: `"mcp_servers[0]: url is required"`. Exit 2. |
449: | Missing transport | `name = "x"`, `url = "..."` but no `transport` | Validation error: transport must be `"sse"` or `"streamable-http"`, got `""`. Exit 2. |
450: | Invalid transport | `transport = "websocket"` | Validation error: transport must be `"sse"` or `"streamable-http"`, got `"websocket"`. Exit 2. |
451: | Duplicate server names | Two entries with `name = "tools"` | Validation error: `"mcp_servers: duplicate server name \"tools\""`. Exit 2. |
452: | No headers | `headers` key absent | `cfg.Headers` is nil. No header injection. |
453: | Empty headers map | `headers = {}` | Empty map. No header injection. |
454: 
455: ### 5.2 Environment Variable Interpolation
456: 
457: | Scenario | Input Value | Env State | Behavior |
458: |----------|-------------|-----------|----------|
459: | Static value | `"Bearer my-token"` | N/A | Returned unchanged. |
460: | Single variable | `"Bearer ${TOKEN}"` | `TOKEN=abc123` | Expanded to `"Bearer abc123"`. |
461: | Multiple variables | `"${SCHEME}://${HOST}"` | Both set | Expanded to full string. |
462: | Missing variable | `"Bearer ${TOKEN}"` | `TOKEN` not set | Error: `"environment variable \"TOKEN\" is not set"`. Exit 2. |
463: | Empty variable | `"Bearer ${TOKEN}"` | `TOKEN=""` | Error: `"environment variable \"TOKEN\" is not set"`. Exit 2. (Empty is treated as unset.) |
464: | Nested braces | `"${FOO${BAR}}"` | N/A | The regex `\$\{([^}]+)\}` matches `${FOO${BAR}` — the first `}` terminates the match. Variable name is `FOO${BAR` which will likely fail lookup. This is not a supported pattern. |
465: | No dollar-brace pattern | `"plain-value"` | N/A | Returned unchanged. |
466: | Dollar without brace | `"$TOKEN"` | N/A | Returned unchanged. Only `${...}` syntax is expanded. |
467: 
468: ### 5.3 MCP Connection
469: 
470: | Scenario | Behavior |
471: |----------|----------|
472: | Server unreachable (connection refused) | `Connect()` returns error. Agent exits with code 3. |
473: | DNS resolution failure | `Connect()` returns error. Agent exits with code 3. |
474: | TLS certificate error | `Connect()` returns error. Agent exits with code 3. |
475: | Server rejects initialize (unsupported protocol version) | `Connect()` returns error. Agent exits with code 3. |
476: | Server responds but `tools/list` fails | `ListTools()` returns error. Agent exits with code 3. |
477: | Server returns zero tools | `ListTools()` returns empty slice. No tools added. Agent proceeds (not an error). |
478: | Connection timeout (exceeds `--timeout`) | Context deadline exceeded. Agent exits with code 3. |
479: 
480: ### 5.4 Tool Name Collisions
481: 
482: | Scenario | Behavior |
483: |----------|----------|
484: | MCP tool `read_file`, built-in `read_file` enabled | MCP tool skipped. Built-in wins. Verbose log emitted. |
485: | MCP tool `read_file`, built-in `read_file` NOT in `cfg.Tools` | MCP tool is NOT skipped. It is registered because the built-in is not active for this agent. `builtinNames` only contains tools from `cfg.Tools`. |
486: | MCP-A has `search`, MCP-B has `search` | Error at startup: `"MCP tool name collision: tool \"search\" is exposed by both \"server-a\" and \"server-b\""`. Exit 2. |
487: | MCP tool `call_agent` | `call_agent` is not in `builtinNames` (it is special-cased, not in the tools registry). The MCP tool named `call_agent` is registered. This is valid — the MCP tool dispatch happens before the special-case `call_agent` check in `executeToolCalls()`. However, this is an unlikely edge case. |

491: | JSON Schema Type | `ToolParameter.Type` | Notes |
492: |-----------------|---------------------|-------|
493: | `"string"` | `"string"` | Direct mapping. |
494: | `"integer"` | `"integer"` | Direct mapping. |

496: | `"boolean"` | `"boolean"` | Direct mapping. |

508: ### 5.6 Tool Call Dispatch
509: 
510: | Scenario | Behavior |
511: |----------|----------|
512: | LLM calls MCP tool by name | Router finds owning client, delegates `CallTool`. Result returned to LLM. |
513: | LLM calls built-in tool by name | Router check returns false. Falls through to registry dispatch. Normal built-in execution. |
514: | LLM calls `call_agent` | Special-cased before router check (existing behavior). Router never consulted. |
515: | LLM calls unknown tool name | Router returns false. Registry dispatch returns error. Error returned to LLM as `ToolResult{IsError: true}`. |
516: | MCP server returns error on `tools/call` | `CallTool` returns error. Wrapped as `ToolResult{IsError: true}`. LLM sees the error message. |
517: | MCP server returns `IsError: true` in result | `ToolResult{IsError: true}` with the text content. |
518: | MCP server returns only ImageContent | `ToolResult{IsError: true, Content: "MCP tool returned no text content"}`. |
519: | MCP server returns mixed TextContent + ImageContent | Only TextContent is extracted. ImageContent ignored. |
520: | MCP server returns multiple TextContent items | All text items concatenated with `"\n"`. |

522: ### 5.7 Sub-Agent MCP
523: 
524: | Scenario | Behavior |
525: |----------|----------|
526: | Parent has MCP, sub-agent has no MCP | Sub-agent runs without MCP tools. Parent's MCP connections are not shared. |
527: | Parent has no MCP, sub-agent has MCP | Sub-agent establishes its own MCP connections, discovers tools, closes on completion. |
528: | Parent and sub-agent both have MCP | Independent connections. No sharing. |
529: | Sub-agent MCP connection fails | Sub-agent returns `ToolResult{IsError: true}`. Parent handles gracefully (existing behavior). |

533: ## 6. Constraints
534: 
535: **Constraint 1:** One new external dependency: `github.com/modelcontextprotocol/go-sdk`.
536: Pin to a specific release version. No other new dependencies.
537: 
538: **Constraint 2:** No changes to `internal/provider/provider.go`. The `Tool`,
539: `ToolCall`, `ToolResult`, and `Message` types are unchanged.
540: 
541: **Constraint 3:** No changes to `internal/toolname/toolname.go`. MCP tool names are
542: not added to `ValidNames()`. MCP tools are not part of the built-in tool system.
543: 
544: **Constraint 4:** `call_agent` remains special-cased outside the registry. MCP tools
545: do not affect the `call_agent` dispatch path.
546: 
547: **Constraint 5:** The `tools` field in agent TOML continues to control only built-in
548: tools. MCP tools are controlled exclusively by `[[mcp_servers]]`.
549: 
550: **Constraint 6:** MCP connections are never shared across agents. Each agent (or
551: sub-agent) establishes and owns its own connections.
552: 
553: **Constraint 7:** Axe is an MCP client only. It does not expose its own tools as an
554: MCP server.
555: 
556: **Constraint 8:** Only `"sse"` and `"streamable-http"` transport types are supported.
557: `"stdio"` is not supported (axe does not launch MCP servers as subprocesses).
558: 
559: **Constraint 9:** No retry logic for MCP connections or tool calls. A single attempt is
560: made. If it fails, the error is returned.
561: 
562: **Constraint 10:** No caching of MCP tool definitions. `tools/list` is called once per
563: connection at startup. Tool definitions are not refreshed during the conversation loop.
564: 
565: **Constraint 11:** The `Headers` field in `MCPServerConfig` applies to all HTTP requests
566: to that MCP server (transport negotiation, initialize, tools/list, tools/call). This is
567: achieved via a custom `RoundTripper` on the `http.Client`.

858: ## 11. Notes
859: 
860: - **Why the official SDK?** The MCP protocol involves JSON-RPC 2.0 framing, SSE parsing,
861:   streamable HTTP content-type negotiation, session ID tracking, and the
862:   initialize/initialized handshake. Implementing this correctly from scratch would be
863:   hundreds of lines of protocol code. The official Go SDK is maintained by the MCP
864:   project team and handles all transport edge cases. The trade-off (one new dependency)
865:   is justified by the complexity avoided.
866: 
867: - **Why built-in wins on collision?** Built-in tools have known, tested behavior within
868:   axe. An MCP server could theoretically expose a tool with the same name but different
869:   semantics. Letting the built-in win prevents surprises. Users who want the MCP version
870:   can remove the built-in from their `tools` list.
871: 
872: - **Why fail-fast on MCP connection failure?** The agent author declared the MCP server
873:   as a dependency. If it's unreachable, the agent cannot fulfill its purpose. Proceeding
874:   without tools leads to silent degradation that's hard to debug. Fail-fast with a clear
875:   error message is more useful.
876: 
877: - **Why no `stdio` transport?** Axe is "executor, not scheduler." The `stdio` transport
878:   requires launching and managing a child process (the MCP server). This adds process
879:   lifecycle management (startup, health, shutdown, signal handling) that conflicts with
880:   axe's "single binary, zero runtime" philosophy. SSE and streamable-HTTP connect to
881:   already-running servers, which aligns with the Unix composition model.
882: 
883: - **Sub-agent isolation.** Each sub-agent establishes its own MCP connections because
884:   sub-agents are opaque to parents (AGENTS.md constraint). Sharing connections would
885:   leak the parent's state to the sub-agent and vice versa. The overhead of extra
886:   connections is acceptable given that axe runs single-purpose agents that exit quickly.
887: 
888: - **JSON Schema flattening.** The flattening strategy (complex types become `"string"`)
889:   works because the LLM reads the tool description and parameter descriptions to
890:   understand expected formats. The `Type` field is a hint, not a strict contract. LLMs
891:   already handle this well — they produce JSON strings that the MCP server can parse.
892: 
893: - **`map[string]string` to `map[string]any`.** Axe's provider abstraction normalizes all
894:   tool arguments to `map[string]string`. This is a simplification that works for all
895:   built-in tools. For MCP tools, converting string values to `any` and letting the MCP
896:   server handle type coercion is the least-invasive approach. Changing the provider
897:   abstraction to `map[string]any` would require touching every provider implementation

2: 
3: **Status:** Draft
4: **Version:** 1.0
5: **Created:** 2026-03-05
6: **GitHub Issue:** https://github.com/jrswab/axe/issues/9
7: **Scope:** MCP client support — connect to external MCP servers, discover tools, and route LLM tool calls to them
8: 
9: ---
10: 
11: ## 1. Purpose
12: 
13: Add support for connecting to MCP servers as a tool source, allowing agents to use tools
14: exposed by external MCP servers alongside axe's built-in tools.
15: 
16: Axe acts as an **MCP client only** (tool consumer). Exposing axe's own tools as an MCP
17: server is out of scope.
18: 
19: An agent declares MCP servers in its TOML config. At startup, axe connects to each
20: declared server, discovers available tools via the MCP `tools/list` method, converts them
21: to `provider.Tool` definitions, and appends them to the LLM request. During the
22: conversation loop, tool calls targeting MCP tools are routed to the appropriate MCP
23: server session via the MCP `tools/call` method.
24: 
25: This feature introduces one new external dependency:
26: `github.com/modelcontextprotocol/go-sdk` (the official MCP Go SDK).
27: 
28: ---
29: 
30: ## 2. Design Decisions
31: 
32: The following decisions were made during planning and are binding for implementation:
33: 
34: 1. **Official Go SDK.** Use `github.com/modelcontextprotocol/go-sdk/mcp` for MCP client
35:    functionality. This provides spec-compliant transport implementations
36:    (`StreamableClientTransport`, `SSEClientTransport`), session management, and
37:    JSON-RPC handling. Pin to a specific release version in `go.mod`.
38: 
39: 2. **Two transport types.** Support `"sse"` and `"streamable-http"` as transport values.
40:    `"sse"` uses the SDK's `SSEClientTransport`. `"streamable-http"` uses the SDK's
41:    `StreamableClientTransport`. No other transport types are supported.
42: 
43: 3. **Eager connection at startup.** All declared MCP servers are connected before the
44:    first LLM call. Tool discovery (`tools/list`) happens immediately after connection.
45:    Sessions remain open for the duration of the conversation loop and are closed when
46:    the agent run completes.
47: 
48: 4. **Fail fast on connection failure.** If any declared MCP server cannot be reached
49:    (connection refused, DNS failure, timeout, TLS error, initialization handshake
50:    failure), the agent fails immediately with exit code 3 (API error).
51: 
52: 5. **Runtime-only tool validation.** MCP tool names are discovered dynamically via
53:    `tools/list`. They are not added to `toolname.ValidNames()`. The `tools` field in
54:    agent TOML controls only built-in tools. MCP tools are controlled entirely by the
55:    `[[mcp_servers]]` configuration.
56: 
57: 6. **Built-in tool wins on name collision.** If an MCP server exposes a tool whose name
58:    matches a built-in tool that is enabled for this agent (present in `cfg.Tools`), the
59:    MCP tool is silently skipped. A warning is logged to stderr in verbose mode. The
60:    built-in tool takes precedence because it has known, tested behavior.
61: 
62: 7. **MCP-to-MCP collision is an error.** If two different MCP servers expose tools with
63:    the same name, the agent fails at startup with exit code 2 (config error). The error
64:    message names both MCP server names and the colliding tool name.
65: 
66: 8. **Flatten complex JSON Schema types.** MCP tools use full JSON Schema for
67:    `InputSchema`. Axe's `provider.ToolParameter` has `Type`, `Description`, and
68:    `Required` fields. Simple JSON Schema types (`string`, `integer`, `number`,
69:    `boolean`) map directly to the `Type` field. Complex types (`object`, `array`,
70:    `oneOf`, `anyOf`, `allOf`, or any type not in the simple set) are mapped to
71:    `Type: "string"` with the full schema description preserved in the `Description`
72:    field. The LLM reads the description and formats accordingly.
73: 
74: 9. **Concatenate text content from results.** MCP `CallToolResult` can contain multiple
75:    content items (`TextContent`, `ImageContent`, `EmbeddedResource`, etc.). All
76:    `TextContent` items are extracted and concatenated with newline (`"\n"`) separators.
77:    Non-text content items are ignored (axe is a text-only CLI tool). If no `TextContent`
78:    items exist in the result, the tool returns `ToolResult{IsError: true}` with content
79:    `"MCP tool returned no text content"`.
80: 
81: 10. **Pass arguments as strings.** Axe's `ToolCall.Arguments` is `map[string]string`.
82:     When dispatching to an MCP server, each string value is placed into a
83:     `map[string]any` as-is (the `any` value is the original `string`). The MCP server is
84:     responsible for any type coercion. This is consistent with how axe handles built-in
85:     tool arguments.
86: 
87: 11. **Environment variable interpolation for headers.** Header values support `${VAR}`
88:     syntax. The pattern `${VAR_NAME}` is replaced with `os.Getenv("VAR_NAME")`. If the
89:     referenced environment variable is not set or is empty, the agent fails at config
90:     resolution time with exit code 2 (config error). The error message names the missing
91:     variable and the MCP server name. Static header values (no `${}` pattern) pass
92:     through unchanged.
93: 
94: 12. **Dry-run shows config only, no connection.** In `--dry-run` mode, MCP server
95:     configuration (name, URL, transport) is displayed in a `--- MCP Servers ---` section.
96:     No connections are established. Tool names are not shown because they require a live
97:     connection.
98: 
99: 13. **Sub-agents get their own MCP connections.** When a sub-agent has `[[mcp_servers]]`
100:     in its own config, it establishes its own MCP connections independently. The parent
101:     agent's MCP connections are never shared with sub-agents. This preserves the
102:     sub-agent opacity principle.
103: 
104: 14. **MCP tool calls appear in `--json` output.** MCP tool calls are included in the
105:     `tool_call_details` array with the same structure as built-in tool calls: `name`,
106:     `input`, `output`, `is_error`. There is no distinction between MCP and built-in tool
107:     calls in the JSON envelope.
108: 
109: 15. **Connection timeout inherits `--timeout`.** MCP server connections use the same
110:     timeout context as the agent run. If the `--timeout` flag is 120 seconds, MCP
111:     connections and tool calls share that deadline.

439: ## 5. Edge Cases
440: 
441: ### 5.1 Config Parsing
442: 
443: | Scenario | Input | Behavior |
444: |----------|-------|----------|
445: | No MCP servers | `mcp_servers` key absent | `cfg.MCPServers` is nil. No MCP logic executes. Agent runs normally. |
446: | Empty MCP servers array | `mcp_servers = []` | Same as absent. Zero-length slice. No MCP logic executes. |
447: | Missing name | `[[mcp_servers]]` with no `name` key | Validation error: `"mcp_servers[0]: name is required"`. Exit 2. |
448: | Missing URL | `name = "x"` but no `url` | Validation error: `"mcp_servers[0]: url is required"`. Exit 2. |
449: | Missing transport | `name = "x"`, `url = "..."` but no `transport` | Validation error: transport must be `"sse"` or `"streamable-http"`, got `""`. Exit 2. |
450: | Invalid transport | `transport = "websocket"` | Validation error: transport must be `"sse"` or `"streamable-http"`, got `"websocket"`. Exit 2. |
451: | Duplicate server names | Two entries with `name = "tools"` | Validation error: `"mcp_servers: duplicate server name \"tools\""`. Exit 2. |
452: | No headers | `headers` key absent | `cfg.Headers` is nil. No header injection. |
453: | Empty headers map | `headers = {}` | Empty map. No header injection. |
454: 
455: ### 5.2 Environment Variable Interpolation
456: 
457: | Scenario | Input Value | Env State | Behavior |
458: |----------|-------------|-----------|----------|
459: | Static value | `"Bearer my-token"` | N/A | Returned unchanged. |
460: | Single variable | `"Bearer ${TOKEN}"` | `TOKEN=abc123` | Expanded to `"Bearer abc123"`. |
461: | Multiple variables | `"${SCHEME}://${HOST}"` | Both set | Expanded to full string. |
462: | Missing variable | `"Bearer ${TOKEN}"` | `TOKEN` not set | Error: `"environment variable \"TOKEN\" is not set"`. Exit 2. |
463: | Empty variable | `"Bearer ${TOKEN}"` | `TOKEN=""` | Error: `"environment variable \"TOKEN\" is not set"`. Exit 2. (Empty is treated as unset.) |
464: | Nested braces | `"${FOO${BAR}}"` | N/A | The regex `\$\{([^}]+)\}` matches `${FOO${BAR}` — the first `}` terminates the match. Variable name is `FOO${BAR` which will likely fail lookup. This is not a supported pattern. |
465: | No dollar-brace pattern | `"plain-value"` | N/A | Returned unchanged. |
466: | Dollar without brace | `"$TOKEN"` | N/A | Returned unchanged. Only `${...}` syntax is expanded. |
467: 
468: ### 5.3 MCP Connection
469: 
470: | Scenario | Behavior |
471: |----------|----------|
472: | Server unreachable (connection refused) | `Connect()` returns error. Agent exits with code 3. |
473: | DNS resolution failure | `Connect()` returns error. Agent exits with code 3. |
474: | TLS certificate error | `Connect()` returns error. Agent exits with code 3. |
475: | Server rejects initialize (unsupported protocol version) | `Connect()` returns error. Agent exits with code 3. |
476: | Server responds but `tools/list` fails | `ListTools()` returns error. Agent exits with code 3. |
477: | Server returns zero tools | `ListTools()` returns empty slice. No tools added. Agent proceeds (not an error). |
478: | Connection timeout (exceeds `--timeout`) | Context deadline exceeded. Agent exits with code 3. |
479: 
480: ### 5.4 Tool Name Collisions
481: 
482: | Scenario | Behavior |
483: |----------|----------|
484: | MCP tool `read_file`, built-in `read_file` enabled | MCP tool skipped. Built-in wins. Verbose log emitted. |
485: | MCP tool `read_file`, built-in `read_file` NOT in `cfg.Tools` | MCP tool is NOT skipped. It is registered because the built-in is not active for this agent. `builtinNames` only contains tools from `cfg.Tools`. |
486: | MCP-A has `search`, MCP-B has `search` | Error at startup: `"MCP tool name collision: tool \"search\" is exposed by both \"server-a\" and \"server-b\""`. Exit 2. |
487: | MCP tool `call_agent` | `call_agent` is not in `builtinNames` (it is special-cased, not in the tools registry). The MCP tool named `call_agent` is registered. This is valid — the MCP tool dispatch happens before the special-case `call_agent` check in `executeToolCalls()`. However, this is an unlikely edge case. |

492: |-----------------|---------------------|-------|

508: ### 5.6 Tool Call Dispatch
509: 
510: | Scenario | Behavior |
511: |----------|----------|
512: | LLM calls MCP tool by name | Router finds owning client, delegates `CallTool`. Result returned to LLM. |
513: | LLM calls built-in tool by name | Router check returns false. Falls through to registry dispatch. Normal built-in execution. |
514: | LLM calls `call_agent` | Special-cased before router check (existing behavior). Router never consulted. |
515: | LLM calls unknown tool name | Router returns false. Registry dispatch returns error. Error returned to LLM as `ToolResult{IsError: true}`. |
516: | MCP server returns error on `tools/call` | `CallTool` returns error. Wrapped as `ToolResult{IsError: true}`. LLM sees the error message. |
517: | MCP server returns `IsError: true` in result | `ToolResult{IsError: true}` with the text content. |
518: | MCP server returns only ImageContent | `ToolResult{IsError: true, Content: "MCP tool returned no text content"}`. |
519: | MCP server returns mixed TextContent + ImageContent | Only TextContent is extracted. ImageContent ignored. |
520: | MCP server returns multiple TextContent items | All text items concatenated with `"\n"`. |

522: ### 5.7 Sub-Agent MCP
523: 
524: | Scenario | Behavior |
525: |----------|----------|
526: | Parent has MCP, sub-agent has no MCP | Sub-agent runs without MCP tools. Parent's MCP connections are not shared. |
527: | Parent has no MCP, sub-agent has MCP | Sub-agent establishes its own MCP connections, discovers tools, closes on completion. |
528: | Parent and sub-agent both have MCP | Independent connections. No sharing. |
529: | Sub-agent MCP connection fails | Sub-agent returns `ToolResult{IsError: true}`. Parent handles gracefully (existing behavior). |

533: ## 6. Constraints
534: 
535: **Constraint 1:** One new external dependency: `github.com/modelcontextprotocol/go-sdk`.
536: Pin to a specific release version. No other new dependencies.
537: 
538: **Constraint 2:** No changes to `internal/provider/provider.go`. The `Tool`,
539: `ToolCall`, `ToolResult`, and `Message` types are unchanged.
540: 
541: **Constraint 3:** No changes to `internal/toolname/toolname.go`. MCP tool names are
542: not added to `ValidNames()`. MCP tools are not part of the built-in tool system.
543: 
544: **Constraint 4:** `call_agent` remains special-cased outside the registry. MCP tools
545: do not affect the `call_agent` dispatch path.
546: 
547: **Constraint 5:** The `tools` field in agent TOML continues to control only built-in
548: tools. MCP tools are controlled exclusively by `[[mcp_servers]]`.
549: 
550: **Constraint 6:** MCP connections are never shared across agents. Each agent (or
551: sub-agent) establishes and owns its own connections.
552: 
553: **Constraint 7:** Axe is an MCP client only. It does not expose its own tools as an
554: MCP server.
555: 
556: **Constraint 8:** Only `"sse"` and `"streamable-http"` transport types are supported.
557: `"stdio"` is not supported (axe does not launch MCP servers as subprocesses).
558: 
559: **Constraint 9:** No retry logic for MCP connections or tool calls. A single attempt is
560: made. If it fails, the error is returned.
561: 
562: **Constraint 10:** No caching of MCP tool definitions. `tools/list` is called once per
563: connection at startup. Tool definitions are not refreshed during the conversation loop.
564: 
565: **Constraint 11:** The `Headers` field in `MCPServerConfig` applies to all HTTP requests
566: to that MCP server (transport negotiation, initialize, tools/list, tools/call). This is
567: achieved via a custom `RoundTripper` on the `http.Client`.

858: ## 11. Notes
859: 
860: - **Why the official SDK?** The MCP protocol involves JSON-RPC 2.0 framing, SSE parsing,
861:   streamable HTTP content-type negotiation, session ID tracking, and the
862:   initialize/initialized handshake. Implementing this correctly from scratch would be
863:   hundreds of lines of protocol code. The official Go SDK is maintained by the MCP
864:   project team and handles all transport edge cases. The trade-off (one new dependency)
865:   is justified by the complexity avoided.
866: 
867: - **Why built-in wins on collision?** Built-in tools have known, tested behavior within
868:   axe. An MCP server could theoretically expose a tool with the same name but different
869:   semantics. Letting the built-in win prevents surprises. Users who want the MCP version
870:   can remove the built-in from their `tools` list.
871: 
872: - **Why fail-fast on MCP connection failure?** The agent author declared the MCP server
873:   as a dependency. If it's unreachable, the agent cannot fulfill its purpose. Proceeding
874:   without tools leads to silent degradation that's hard to debug. Fail-fast with a clear
875:   error message is more useful.
876: 
877: - **Why no `stdio` transport?** Axe is "executor, not scheduler." The `stdio` transport
878:   requires launching and managing a child process (the MCP server). This adds process
879:   lifecycle management (startup, health, shutdown, signal handling) that conflicts with
880:   axe's "single binary, zero runtime" philosophy. SSE and streamable-HTTP connect to
881:   already-running servers, which aligns with the Unix composition model.
882: 
883: - **Sub-agent isolation.** Each sub-agent establishes its own MCP connections because
884:   sub-agents are opaque to parents (AGENTS.md constraint). Sharing connections would
885:   leak the parent's state to the sub-agent and vice versa. The overhead of extra
886:   connections is acceptable given that axe runs single-purpose agents that exit quickly.
887: 
888: - **JSON Schema flattening.** The flattening strategy (complex types become `"string"`)
889:   works because the LLM reads the tool description and parameter descriptions to
890:   understand expected formats. The `Type` field is a hint, not a strict contract. LLMs
891:   already handle this well — they produce JSON strings that the MCP server can parse.
892: 
893: - **`map[string]string` to `map[string]any`.** Axe's provider abstraction normalizes all
894:   tool arguments to `map[string]string`. This is a simplification that works for all
895:   built-in tools. For MCP tools, converting string values to `any` and letting the MCP
896:   server handle type coercion is the least-invasive approach. Changing the provider
897:   abstraction to `map[string]any` would require touching every provider implementation
