25: ### Current Behavior
26: 
27: 1. **HTTP client:** `urlFetchExecute` calls `http.DefaultClient.Do(req)` — no per-request timeout. The only timeout protection is the parent `ctx` passed from the agent runner (default 120s).
28: 2. **Response handling:** The raw response body is returned verbatim regardless of `Content-Type`. For `text/html` responses, this means full HTML markup including `<script>`, `<style>`, navigation, ads, etc. are sent to the LLM.
29: 3. **Truncation:** Body is read via `io.LimitReader(resp.Body, maxReadBytes+1)` where `maxReadBytes = 10000`. If the body exceeds 10,000 bytes, it is truncated with a notice appended.
30: 4. **URL sanitization:** Credentials and query strings are stripped from verbose logs via `sanitizeURL()`.
31: 
32: ### Key Decisions Made
33: 
34: | Decision | Rationale |
35: |----------|-----------|
36: | **15-second per-request timeout** | Matches the TypeScript implementation's `AbortSignal.timeout(15000)`. Long enough for slow servers, short enough to not block an agent run. |
37: | **Use `golang.org/x/net/html` for HTML stripping** | User chose DOM-based parsing over regex. Handles malformed HTML, nested scripts, and edge cases correctly. Adds one dependency (`golang.org/x/net`) but it's a well-maintained Go sub-repo. |
38: | **Strip `<script>` and `<style>` elements (including children)** | These elements contain code/CSS that is noise for LLMs. The TypeScript implementation strips these same elements. |
39: | **Detect `text/html` via `Content-Type` header** | Only strip HTML when the server declares the response as HTML. Non-HTML responses (JSON, plain text, XML) are returned verbatim even if they contain angle brackets. |
40: | **Truncation applies after stripping** | Stripping reduces content size, so truncating after stripping maximizes useful content within the 10,000 character limit. |
41: | **`maxReadBytes` (10,000) unchanged** | The existing limit matches the TypeScript implementation and is sufficient for LLM consumption. |
42: 
43: ### Approaches Considered & Rejected
44: 
45: | Approach | Why Rejected |
46: |----------|-------------|
47: | **Regex-based HTML stripping** (`regexp` stdlib) | User explicitly chose DOM parsing. Regex is fragile with malformed HTML, can't properly handle nested `<script>` tags, and fails on edge cases like `<script>` inside attribute values. |
48: | **Custom HTTP client with transport-level timeout** | Overly complex. `context.WithTimeout` on the request context is the idiomatic Go approach and achieves the same result with less code. |
49: | **Configurable timeout per agent** | Out of scope for this issue. The 15-second default matches the TypeScript implementation. Can be added later if needed. |
50: | **Using `x/net/html` `Tokenizer` (streaming)** | The full `html.Parse` tree approach is simpler to reason about for element skipping (script/style) and produces cleaner code. Performance is not a concern at 10KB max input. |
51: 
52: ### Constraints & Assumptions
53: 
54: - **No new public API.** All changes are internal to the `tool` package. `stripHTML` is unexported.
55: - **Existing tests must pass unchanged.** The 13 existing tests cover: success, empty URL, missing URL, unsupported schemes, HTTP error codes, truncation, context cancellation, connection refused, CallID passthrough, empty body, verbose log sanitization. None of these serve `text/html` content, so they are unaffected by HTML stripping.
56: - **The existing `TestURLFetch_ContextCancellation` test** uses a 50ms parent timeout. Since `min(50ms, 15s) = 50ms`, the parent timeout still wins — this test continues to pass with the new 15-second per-request timeout.
57: - **Red/green TDD required.** Tests are written first (red), then implementation (green), per project rules.
58: - **`golang.org/x/net`** is the only new dependency. It's a Go official sub-repository, widely used, and already transitively available via `golang.org/x/oauth2` in the dependency tree.
59: - **Whitespace collapsing.** After extracting text nodes from the DOM, consecutive whitespace should be collapsed to single spaces and the result trimmed. This prevents large runs of blank lines from wasting the character budget.

63: | # | Question | Answer |
64: |---|----------|--------|
65: | 1 | Should the timeout be configurable via TOML? | **No.** Out of scope. The 15-second hardcoded default matches the TypeScript implementation. Can be revisited in a future issue. |
66: | 2 | Regex or DOM parser for HTML stripping? | **DOM parser** (`golang.org/x/net/html`). User explicitly chose this approach for correctness with malformed HTML. |
67: | 3 | Should `<noscript>` content be preserved? | **Yes.** Only `<script>` and `<style>` are stripped. `<noscript>` often contains meaningful fallback text. |
68: | 4 | What about `Content-Type: text/html; charset=utf-8`? | The check uses `strings.HasPrefix` (or `mime.ParseMediaType`) on the Content-Type header, so charset parameters are handled correctly. |
69: | 5 | Should HTML stripping apply to error responses (non-2xx)? | **Yes.** Error pages are often HTML too. Stripping before returning the error body keeps error messages clean. |

26: 
27: - `urlFetchExecute` creates an HTTP request via `http.NewRequestWithContext(ctx, ...)` and executes it with `http.DefaultClient.Do(req)`.
28: - There is **no per-request timeout**. The only timeout protection is the parent `ctx` passed from the agent runner (default 120 seconds).
29: - A slow server that trickles data or delays the response will block the agent for up to 120 seconds before the parent context cancels.
30: 
31: ### Decisions Already Made
32: 
33: | Decision | Rationale |
34: |----------|-----------|
35: | **15-second hardcoded timeout** | Matches the TypeScript implementation's `AbortSignal.timeout(15000)`. Long enough for slow servers, short enough to not block an agent run. |
36: | **Use `context.WithTimeout` on the request context** | Idiomatic Go. Wraps the incoming `ctx` with a 15-second deadline. The shorter of the two deadlines (parent vs. per-request) wins automatically. |
37: | **Timeout is NOT configurable via TOML** | Out of scope for this issue. Can be revisited in a future issue. |
38: 
39: ### Approaches Ruled Out — Do Not Re-Evaluate
40: 
41: | Approach | Why Rejected |
42: |----------|-------------|
43: | Custom HTTP client with transport-level timeout | Overly complex. `context.WithTimeout` achieves the same result with less code. |
44: | Configurable timeout per agent | Out of scope. |
45: 
46: ### Constraints
47: 
48: - **No new public API.** All changes are internal to the `tool` package.
49: - **All 13 existing tests must pass unchanged.** None of them are affected by a 15-second timeout because:
50:   - Most tests use `httptest.NewServer` which responds instantly.
51:   - `TestURLFetch_ContextCancellation` uses a 50ms parent timeout. Since `min(50ms, 15s) = 50ms`, the parent timeout still wins.
52:   - `TestURLFetch_ConnectionRefused` fails at TCP connect, before any timeout applies.
53: - **Red/green TDD required.** Write failing tests first, then implement.
54: - **No new dependencies.** `context.WithTimeout` is stdlib.

58: ## Section 2: Requirements
59: 
60: ### R1: Per-Request Timeout
61: 
62: When `urlFetchExecute` makes an HTTP request, the request MUST be governed by a 15-second timeout **in addition to** the parent context's deadline.
63: 
64: - The timeout starts when the HTTP request begins (before TCP connect).
65: - The timeout covers the entire request lifecycle: DNS resolution, TCP connect, TLS handshake, sending the request, and reading the response headers.
66: - If the 15-second timeout fires before the parent context's deadline, the request MUST be cancelled and an error returned.
67: - If the parent context's deadline fires before the 15-second timeout, the request MUST be cancelled and an error returned (existing behavior, unchanged).
68: - The shorter deadline always wins. This is the natural behavior of nested Go contexts.
69: 
70: ### R2: Timeout Constant
71: 
72: The 15-second timeout value MUST be defined as a package-level constant (unexported) for clarity and testability. It MUST NOT be a magic number inline.
73: 
74: ### R3: Error Behavior on Timeout
75: 
76: When the per-request timeout fires:
77: 
78: - The result MUST have `IsError: true`.
79: - The result MUST contain the error message from the context cancellation (Go's `context.DeadlineExceeded` surfaces through `http.Client.Do`).
80: - The `CallID` MUST be set to the incoming `call.ID` (existing behavior, unchanged).
81: - The verbose log MUST still fire via the existing `defer` block (existing behavior, unchanged).
82: 
83: ### R4: No Change to Existing Behavior for Fast Responses
84: 
85: Responses that complete within 15 seconds MUST behave identically to today. No observable difference in output, error handling, truncation, or verbose logging.
86: 
87: ### R5: No Change to `maxReadBytes` or Truncation Logic
88: 
89: The timeout governs the HTTP request lifecycle only. The existing `maxReadBytes` (10,000) limit and truncation logic are unrelated and MUST NOT change.
90:

92: 
93: ### Edge Cases
94: 
95: | Case | Expected Behavior |
96: |------|-------------------|
97: | Server responds in 14.9 seconds | Success. Response returned normally. |
98: | Server responds in 15.1 seconds | Error. Per-request timeout fires. `IsError: true`. |
99: | Parent context has 5-second deadline, server responds in 6 seconds | Error. Parent context fires first (existing behavior). |
100: | Parent context has 5-second deadline, server responds in 4 seconds | Success. Both deadlines are satisfied. |
101: | Parent context has no deadline, server never responds | Error. Per-request 15-second timeout fires. |
102: | Server sends headers quickly but trickles body over 20 seconds | The timeout covers the full request. If `http.Client.Do` returns before the timeout (it returns after headers), the body read via `io.ReadAll` is governed by the same context. The request is cancelled at 15 seconds. |
103: | Connection refused (TCP RST) | Error returned immediately, before any timeout. Unchanged behavior. |
104: | DNS resolution fails | Error returned immediately or quickly, before the 15-second timeout. Unchanged behavior. |
105: 
106: ---
107: 
108: ### Parallelism Notes
109: 
110: This milestone is a single, focused change. There are no independent sub-tasks that benefit from parallel execution. The test(s) and implementation are sequential (red/green TDD).
111: 
112: However, this milestone (001) is **independent of milestone 002** (HTML stripping). They can be developed on the same branch sequentially but do not depend on each other.

64: - The timeout starts when the HTTP request begins (before TCP connect).
65: - The timeout covers the entire request lifecycle: DNS resolution, TCP connect, TLS handshake, sending the request, and reading the response headers.
66: - If the 15-second timeout fires before the parent context's deadline, the request MUST be cancelled and an error returned.
67: - If the parent context's deadline fires before the 15-second timeout, the request MUST be cancelled and an error returned (existing behavior, unchanged).
68: - The shorter deadline always wins. This is the natural behavior of nested Go contexts.
69: 
70: ### R2: Timeout Constant
71: 
72: The 15-second timeout value MUST be defined as a package-level constant (unexported) for clarity and testability. It MUST NOT be a magic number inline.
73: 
74: ### R3: Error Behavior on Timeout
75: 
76: When the per-request timeout fires:
77: 
78: - The result MUST have `IsError: true`.
79: - The result MUST contain the error message from the context cancellation (Go's `context.DeadlineExceeded` surfaces through `http.Client.Do`).
80: - The `CallID` MUST be set to the incoming `call.ID` (existing behavior, unchanged).
81: - The verbose log MUST still fire via the existing `defer` block (existing behavior, unchanged).
82: 
83: ### R4: No Change to Existing Behavior for Fast Responses
84: 
85: Responses that complete within 15 seconds MUST behave identically to today. No observable difference in output, error handling, truncation, or verbose logging.
86: 
87: ### R5: No Change to `maxReadBytes` or Truncation Logic
88: 
89: The timeout governs the HTTP request lifecycle only. The existing `maxReadBytes` (10,000) limit and truncation logic are unrelated and MUST NOT change.
